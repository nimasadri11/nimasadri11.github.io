## The Implicit Spec

There is no RFC for agent CLIs. No one published "Attention Is All You Need" for tool-using agents. But two of the largest AI labs built the same system from scratch, and the architecture they converged on tells us something about what an agent CLI *has* to be.

OpenAI shipped Codex. Anthropic shipped Claude Code. The teams never coordinated. They started from different languages, different models, different design philosophies. They arrived at the same architecture.

This post presents that architecture as if it were a paper: the spec nobody wrote down but both teams implemented. Each section blockquotes a piece of the spec, then annotates it with real code from both systems. Like the Annotated Transformer, but for agent tooling, and with two implementations instead of one.

## The Architecture

Every agent CLI is a loop. Take a natural language instruction, send it to a language model, execute the tool calls the model requests, repeat until the task is done. The interesting engineering is everything around that loop. Eleven components hold it together.


> **1. The Session.** An agent CLI must maintain a stateful session that outlives any single model call. The session owns the conversation history, tracks configuration (which model, which tools, which safety rules), and provides the coordination point between the user interface, the model, and the tool executor. Without a session abstraction, there is no way to persist context across turns or recover from failures mid-task.

The session is the spine. Everything else hangs off it.

> **2. The Prompt Builder.** Before each model call, the system must assemble a prompt from multiple sources: system instructions that define the agent's behavior, tool definitions that tell the model what it can do, conversation history from previous turns, and any user-provided context like file contents or project rules. The prompt builder's job is to merge these sources into a single payload that fits within the model's context window, applying caching strategies to minimize redundant tokens.

This is where the two systems diverge hardest. Same inputs, very different assembly strategies.

> **3. The Stream.** The system must call the model's API and process the response as a stream, not a batch. Streaming lets the user see progress in real time and, more importantly, lets the tool executor begin parsing tool calls before the full response arrives. The stream must handle connection failures, rate limits, and partial responses without losing state.

Streaming is table stakes for UX. But streaming *with tool parsing* is where the real complexity lives.

> **4. The Tool Executor.** When the model's response contains tool calls, the executor must parse them, dispatch each to the appropriate handler, and return results to the conversation. Some tool calls are independent and can run in parallel. Others depend on shared state and must be serialized. The executor must handle tool failures gracefully, returning useful error messages to the model rather than crashing the session.

The tool executor is the agent's hands. Get it wrong and the model can't do anything useful.

> **5. The Security Layer.** Every tool call is a potential code execution on the user's machine. The security layer must decide, for each call, whether to allow it, deny it, or ask the user for permission. This requires a policy system flexible enough to express rules like "allow all file reads, but ask before any shell command that modifies files." The security layer must be fast enough that it doesn't break the flow of an interactive session.

This is where the philosophical split runs deepest. Sandbox the environment, or gate each action?

> **6. The Loop Controller.** The outer loop must decide, after each model response, whether to continue or stop. It must detect when the model has finished its task, when it's stuck in a cycle, when the context window is about to overflow, and when an unrecoverable error has occurred. It must handle recovery strategies: compacting the conversation, falling back to a different model, or retrying with backoff.

The loop controller is the least glamorous component and the one that determines whether the agent finishes the job or dies halfway through.

> **7. The Terminal UI.** An agent CLI needs to render streaming text, display code diffs, show approval dialogs, and accept user input, all in a terminal. Two paradigms dominate: immediate-mode rendering (draw every frame from state) and retained-mode rendering (describe what the UI should look like, let a framework diff and patch).

The terminal is the oldest interface in computing, and it turns out to be a hard rendering target when you're streaming markdown at hundreds of tokens per second.

> **8. The Extension Layer.** A useful agent CLI must be extensible beyond what its creators anticipated. Users need to hook into the agent's lifecycle, connect external tool servers via MCP, define reusable behaviors as skills, and install third-party plugins.

The extension surface is what separates a tool from a platform.

> **9. The Swarm.** When a task is too large for one agent, the system needs to coordinate multiple agents working in parallel. This requires decisions about communication topology, permission delegation, physical isolation, and background task management.

Spawning one child agent is Act 5. Running five in parallel, each needing permissions and producing output, is a different problem entirely.

> **10. The Platform Layer.** Beyond the core loop, each team built features the other didn't. These unique capabilities reveal each team's theory of what an agent CLI should become.

Codex pushed downward toward the operating system. Claude Code pushed outward toward the developer's workflow. Both are right about the future.

> **11. The Memory Layer.** A stateless agent loop forgets everything between sessions. A useful agent remembers: what the user prefers, what the codebase looks like, what worked last time.

An agent that forgets everything between sessions is a fancy autocomplete. Memory is what separates an agent that learns from one that merely executes.

## How to Read This Post

Ten acts, each covering a component of the architecture. Acts 1 through 5 cover the core loop: bootstrapping a session, building prompts, streaming responses, gating tool calls, and controlling the outer loop. Acts 6 through 10 cover the surrounding systems: terminal rendering, extensibility, multi-agent coordination, platform-specific features each team built that the other didn't, and memory that persists across sessions. Each act shows pseudocode from both systems and ends with verdicts on what we'd steal from each. Start at Act 1 or jump to any decision that interests you.
